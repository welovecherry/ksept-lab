# 🧪 실험 런북 (Experiments Runbook)

> STRATEGY.md의 §8.5~8.7(실험 자동화 하네스)의 **실행 절차편**. 전략은 STRATEGY.md, *어떻게 돌릴지*는 여기.
> 핵심 원칙: **전부 조합(full grid) ❌ → 한 번에 하나씩(탐욕적 순차) ✅**, **싼 검색부터 → 비싼 생성은 마지막**.

---

## 0. 용어 (먼저 쉽게)

- **베이스라인(baseline)** = 비교의 출발점. 아무것도 안 고친 현재 설정의 점수.
- **축/손잡이(knob)** = 바꿀 수 있는 설정(청크 크기·임베딩·검색 방식 등).
- **OFAT / 탐욕적 순차** = 손잡이를 **하나만** 돌리고 나머지는 고정 → 이긴 값을 *고정한 채* 다음 손잡이로.
- **Recall@K(회수율)** = *정답 §조항이 검색 상위 K개 안에 들어온 비율*. 검색 품질을 **Claude 없이 코드로** 채점 → 공짜.
- **홀드아웃(hold-out)** = 튜닝에 안 쓰고 검증에만 쓰는 *비밀 시험지*. → `holdout.jsonl`.

## 1. 왜 "전부 조합"이 아니라 "순차"인가

축 8개를 전부 조합하면 3×3×4×3×3×2×2×2 = **약 5천 가지** → 호출·예산·시간 폭발.
**탐욕적 순차**는 손잡이를 하나씩 최적화(=축 개수만큼만 실험) → 해석도 쉽고 호출도 적다.

```mermaid
flowchart LR
    B[베이스라인] --> E1[실험1 청킹] -->|이긴 값 고정| E2[실험2 임베딩]
    E2 -->|고정| E3[실험3 검색] -->|고정| E4[실험4 topK]
    E4 -->|고정| E5[실험5 모델] -->|고정| E6[실험6 프롬프트]
```

> 한계 인정: 순차 탐색은 *손잡이끼리 상호작용*이 크면 진짜 최적을 놓칠 수 있다. 그래도 6시간·예산 제약에선 **가성비 최고의 1순위 전략**. 여유가 남으면 상위 2~3개 설정 주변만 좁은 그리드로 재확인.

> #### 🔴 [비판적 수정 1] "전부 조합 ❌"는 **유료 축에만** — 무료 축(1~4)은 넓게 그리드
> 순차(OFAT)를 쓰는 *유일한 이유*는 **호출·예산 폭발**인데, **실험 1~4는 Claude를 안 부르니 돈이 0**이다. 폭발하는 건 *유료 축(모델·프롬프트)*뿐. 무료 검색 축만 곱하면 청킹3 × 임베딩4 × 검색3 × K3 = **108개 — 하룻밤에 충분히 다 돈다.**
> - **권장:** 검색 축(1~4)은 **부분 전조합(작은 그리드)**, 생성 축(5~6)만 **탐욕적 순차**.
> - **왜?** OFAT는 *손잡이 상호작용*을 못 본다. 예: **청크가 작아지면 한 조각의 정보가 적어 top-K를 키워야** 같은 Recall이 나온다(청킹↔K 상호작용). 무료인데 굳이 이 위험을 떠안을 이유가 없다.

## 2. 실험 순서 (싸고 효과 큰 것 먼저)

| # | 실험(축) | 후보 | 채점 | Claude 호출 |
|---|---------|------|------|:---:|
| 0 | **베이스라인** | 현재(1000자·MiniLM·의미·K5·sonnet) | Recall@K + (선택)심판 | 최소 |
| 1 | 청킹 | 1000자 / §경계 / 문서별 라우팅 | **Recall@K** | ❌ |
| 2 | 임베딩 모델 | 기준=MiniLM · 후보=bge-large-en-v1.5 / e5-large-v2 / gte-large (※bge·e5는 쿼리 프리픽스 필수, gte는 불필요) | **Recall@K** | ❌ |
| 3 | 검색 방식 | 의미 / BM25 / 하이브리드(α) | **Recall@K** | ❌ |
| 4 | top-K | 3 / 5 / 8 | Recall@K + 토큰비용 | ❌ |
| 5 | **생성 모델** | sonnet-4-6 / opus-4-8 | **LLM 심판 + 인용검증** | ✅(마지막) |
| 6 | 프롬프트 | 종합·거부 강조 A/B | **LLM 심판** | ✅ |

`★ 핵심:` 실험 1~4는 **검색 단계만** 보므로 *Claude를 안 부르고 Recall@K로 공짜 채점*. 비싼 생성 호출은 검색이 확정된 **실험 5~6에만**. → 예산 1/10.

> #### 🔴 [비판적 수정 2] 루프 중첩으로 **재인덱싱 횟수**를 108 → 12로
> 같은 무료라도 비용이 다르다. **청킹·임베딩을 바꾸면 인덱스를 다시** 만들어야(임베딩 재계산·모델 다운로드) 하지만, **검색 방식·top-K는 같은 인덱스에서 쿼리만** 바꾸면 된다(메모리에서 즉시).
> ```
> for 청킹 in [...]:            # 바깥: 무거운 재인덱싱
>   for 임베딩 in [...]:        #   (청킹×임베딩 = 12개 인덱스만 빌드)
>     index = build(청킹, 임베딩)
>     for 검색 in [...]:        # 안쪽: 같은 인덱스 재사용 → 공짜·즉시
>       for K in [...]:         #   (검색×K = 9조합을 메모리에서 스윕)
>         eval_recall(index, 검색, K)
> ```
> → 인덱스 빌드 **12번**으로 108개 검색설정 평가. 야간 시간이 확 준다.

> #### 🔴 [비판적 수정 3] 검색 1등 ≠ 생성 1등 → **상위 2개**를 생성으로 올려라
> Recall@K가 가장 높은 설정이 *답·인용도 1등*이란 보장은 없다(다양성 높은 설정이 종합에 유리할 수 있음). 검색 단계에서 **finalist 2개**를 남겨 실험 5~6에 둘 다 태우고 거기서 최종 결정. 그리디 체인이 단일 승자를 너무 일찍 못박는 걸 막는 안전장치.

## 3. "한 번의 실험"은 이렇게 돈다

```
1. 설정 고정 (예: §경계 + bge + 하이브리드 + K=5)
2. holdout.jsonl 의 질문 전부를 그 설정으로 실행
3. 채점:
   - 검색 → Recall@K        (코드, 무료)   : 정답 §가 top-K에 있나?
   - 생성 → LLM 심판 1~5점  (실험 5~6만)  : 사실성·인용·완전성
   - 인용 → 프로그램 검증    (코드, 무료)   : 인용한 §가 실제 그 청크에 있나?
4. 평균 점수를 runs.jsonl 한 줄로 append
5. 다음 설정으로 반복 → 이긴 값 고정
```

## 4. 채점 방법 3종

1. **Recall@K (무료·객관)** — `holdout.jsonl`의 `expected_sections`가 검색된 top-K 청크의 §메타에 포함되면 1, 아니면 0. 평균이 그 설정의 검색 점수.

> #### 🔴 [비판적 수정 4] 이진 Recall은 거칠다 → **집합 커버리지 + 순위(MRR)** 로 보강
> ① **정답은 "집합"이다.** 교차질문(H06·H10·H11)은 정답 §이 여러 개 → "하나라도 있으면 1"인 이진 Recall은 *절반만 맞아도 만점*이라 거짓 신호. **coverage = (top-K에 든 정답 § 수) ÷ (정답 § 총수)** 로 재라. (`expected_sections`를 항상 *배열*로 다루고, 단일정답도 길이 1 집합으로.)
> ② **순위도 본다(MRR).** 정답이 1위냐 8위냐는 토큰·리랭킹·생성품질에 영향. **MRR**(정답 첫 등장 순위의 역수, 1위=1.0·5위=0.2)을 한 칸 더 기록.
> ③ **작은 N 노이즈.** 12문제면 1문제 = 8.3%. **근소한 차(1문제)는 노이즈로 간주**, 동점이면 *더 싸고 단순한* 설정을 택한다.
2. **LLM 심판 (3막·Claude Code)** — 루브릭으로 사실성/인용정확성/완전성 각 1~5점. 생성 모델과 분리(심판=opus).
3. **프로그램 인용검증 (무료)** — 답변이 인용한 §번호의 청크 텍스트에 그 주장 단어가 실제로 있는지 대조 → 가짜 인용 색출.

> 범위밖·적대형 질문은 별도 채점: **"거부했는가"가 정답**(`expect_refusal: true`).

## 5. 비용 가드 (공유 키 $100)

> **방침:** 무료(로컬 검색)는 **마음껏 많이** 돌린다. 아끼는 건 오직 **공용 API(생성) 비용**뿐. → 검색 축은 넓게, 절약은 생성 쪽에서만.

- 검색 실험(1~4)은 **$0** (로컬 임베딩).
- 생성 실험(5~6)만 과금 → **Batches API 50%↓** + 누적 상한 가드(초과 시 자동 중단).
- 심판은 **Claude Code(공유키 0원)**.

### 5.1 토큰이 진짜 비용 단위 (호출 수 ≠ 비용)

과금은 **토큰**(입력+출력)으로 매겨진다. 호출 수가 적어도 호출당 토큰이 크면 비싸다. 출력 토큰 단가는 입력의 ~4~5배.

- **[적용 ①] 프롬프트 캐싱 필수** — 생성 호출의 *고정 부분*(시스템 지시문·인용 규칙)에 `cache_control`을 붙여 재사용. 밤샘에 같은 시스템 프롬프트를 수십 번 반복하므로 두 번째 호출부터 그 부분이 ~10% 값. 라이브(`app.py`)·하네스 생성 함수 **양쪽 모두**.
- **[적용 ②] 적정 top-K = 실험 4의 산출물** — 검색 문맥(top-K 청크)이 생성 입력 토큰의 대부분. 그래서 실험 4는 Recall@K뿐 아니라 **호출당 토큰 비용을 함께 기록**하고, **Recall이 사실상 동률이면 토큰이 적은(작은 K) 설정을 택한다**. "무료 검색에서 내린 K가 유료 토큰으로 흘러든다"는 점을 명시.
- (보조) 답변 간결화(`max_tokens` 상한 + 프롬프트에 "핵심만"), §경계 청킹으로 군더더기 최소화.
- **runs.jsonl에 `tokens.in`·`tokens.out`을 항상 기록** — K·캐싱의 실효를 사후 검증하려면 토큰이 데이터로 남아야 함.

> #### 🔴 [비판적 수정 5] 정답 § 검증은 선택이 아니라 **차단 작업(blocker)**
> 이 전체 무료 채점이 **`expected_sections` 라벨의 정확성에 통째로 의존**한다. 라벨이 틀리면 Recall@K가 거짓말을 하고 *틀린 방향으로 빠르게* 달린다. 지금 holdout 다수가 `"추정 — 확인"` 상태(예: H05 최소나이를 §61.103으로 추정 — 실제 확인 필요).
> - **검증 방법(Phase 1 직후, 코드로):** FAA 텍스트 추출 후 `grep`으로 *정답 § 청크에 질문의 핵심 키워드가 실제로 있는지* 대조. 예: H03 → `§91.151` 청크에 "30 minutes"·"fuel" 있나.
> - **검증 끝나기 전엔 실험 1~4 점수를 신뢰하지 말 것.** ← 이게 안 끝나면 야간 그리드가 *쓰레기를 빠르게* 만든다.

> #### 🔴 [비판적 수정 6] 홀드아웃 **자기 과적합** 방지
> 12~15문제에 검색을 빡세게 맞추면 *그 세트에만* 강해질 수 있다(§9 과적합과 같은 함정). **3~4문제는 "최종 점검용"으로 떼어 튜닝에 안 쓰고** 맨 마지막 한 번만 확인. 연습문제 5개도 *최적화 대상이 아니라 sanity check*로만.
