# 1막 (B) 하네스 — 정답지 검증 → 채점기 → 오케스트레이터 → 인용검증

> 🎬 **1막 = 파이프라인 고도화 + 실험 하네스 제작.** 이 문서는 그 중 **(B) 하네스**. **(A) 파이프라인**(단계 1~5, ✅완료)은 → [`06_30_act1_pipeline.md`](06_30_act1_pipeline.md).
> 하네스가 완성돼야 **2막(밤샘 자동실험)** 을 돌릴 수 있다. 전략=[STRATEGY.md](../STRATEGY.md), 실행 런북=[EXPERIMENTS.md](../EXPERIMENTS.md).

## 사용자 시나리오 (가장 먼저 읽힌다)

- **누가**: 홍정민 — 대회 전날, "어떤 검색 설정이 제일 잘 맞히나"를 *밤새 무인으로* 알아낼 **채점 기계**를 만드는 사람.
- **언제**: 파이프라인(§인용까지)이 도는 걸 확인한 직후, 아직 *어떤 청킹·임베딩·검색법이 최적인지 모르는* 순간.
- **어떻게**:
  1. 먼저 **정답지(holdout 14문항)가 진짜 맞는지** 교재(FAA)로 대조해 확정한다. ← 안 하면 나머지가 전부 거짓말.
  2. "정답 §가 검색 top-K에 들었나"를 **Claude 없이 코드로** 채점하는 자를 만든다(Recall·coverage·MRR).
  3. 여러 설정을 순서대로 도는 **for문 엔진(오케스트레이터)** 을 만들어, 한 실행마다 `runs.jsonl` 한 줄을 남긴다.
  4. 답변이 인용한 §가 *진짜 그 근거 청크에 있는지* 대조하는 **인용검증기**를 만든다.
- **결과**: 밤새 돌릴 수 있는 **채점 하네스 완성** → 2막에서 "그냥 켜두면 아침에 최적 설정이 나와 있는" 상태.
- **잘 안 될 때(실패 시 보이는 것)**:
  - 정답지가 틀리면 → Recall이 *거짓말*을 하고 야간 그리드가 "쓰레기를 빠르게" 만든다([🔴수정5]).
  - 채점이 이진(있다/없다)뿐이면 → 교차질문(정답 § 여러 개)이 *절반만 맞아도 만점*으로 뜬다([🔴수정4]).

> 비유: **자율학습 로봇에게 "정답지 + 채점기 + 시험 진행표"를 쥐여주는** 작업. 로봇이 밤새 혼자 모의고사를 풀고 스스로 채점하게.

## 배경

- (A) 파이프라인으로 **FAA 인덱스(2184청크, §·part 메타)** 와 **기록 로거(runs.jsonl)** 는 이미 있다.
- 하지만 EXPERIMENTS.md §6 체크리스트의 나머지 — **정답 § 검증·채점기·오케스트레이터·인용검증** — 이 없어 아직 *측정*을 못 한다.
- 이 하네스가 쓰는 채점은 전부 **무료 로컬**(Recall@K·coverage·MRR·인용대조). 유료 LLM 심판은 3막.

### 📂 파일 지도 — `.jsonl` 3종은 서로 다르다 (자주 헷갈림)

`.jsonl`은 *"한 줄에 하나씩 적는 파일 형식"* 일 뿐. **그 형식에 무엇을 담느냐**가 완전히 다르다:

| 파일 | 무엇을 담나 | 누가/언제 | 변하나? |
|---|---|---|:---:|
| **holdout.jsonl** | **정답지** (14문제 + 각 정답 §) | 사람이 만들고 확정 | ❌ 고정 (기준) |
| **runs.jsonl** | **실험 결과** (설정별 채점 점수) | 로봇이 밤새 append | ✅ 계속 쌓임 |
| **index_manifest.jsonl** | **인덱스 빌드 기록** | 인덱서가 빌드마다 | ✅ 쌓임 |

> 핵심: **정답지(holdout)** 와 **실험기록(runs)** 은 *다른 파일*이다. 채점기(H2)가 holdout.jsonl을 열어 "정답 §"를 읽고, 검색결과와 대조한 점수를 runs.jsonl에 남긴다.

> 관심사: *"정답지가 진짜 맞나."* — 채점기보다 먼저. 정답지가 틀리면 모든 점수가 거짓.

#### 🟢 쉬운 설명

지금 정답지 14개 중 **11개가 "아마 이게 정답일걸?"(추정)**, 3개(H01·H02·H06)는 **정답 칸이 비어** 있다. 로봇을 돌리기 전에 정답지부터 확정한다.
- **추정 확인** = 교재에서 진짜 찾아보기: 예) H03("VFR 연료?") 정답을 §91.151로 적었으면, *진짜 §91.151 청크에 "30 minutes"·"fuel"이 있는지* 대조. 있으면 ✅, 없으면 수정.
- **빈 칸 채우기** = 이미 만든 `search()`로 질문을 던져 후보 §를 찾아 채움.

> **📂 H1이 만지는 건 정답지(holdout.jsonl)** — 단계 2의 실험기록(runs.jsonl)과 *다른 파일*이다. (`.jsonl` 3종 구분은 위 [배경 §파일 지도](#-파일-지도--jsonl-3종은-서로-다르다-자주-헷갈림) 참조.)

> **💡 왜 "정답지 확정"이 먼저인가 (비유: 채점 전 정답지 검사).** 원래 정답지는 *대충 기억으로* 적혀 있었다("VFR 연료? 아마 §91.151"). 그 정답이 틀리면 로봇이 잘 풀어도 *틀리게 채점*된다. 그래서 ① 각 답을 교재(FAA)에서 진짜 찾아 확인(§91.151에 "day 30분" 실재 → ✅), ② 빈 칸은 교재를 뒤져 채움(H02 "category 정의?" → §1.1에 실재). 정답지가 "추정"→"확인됨"이 되어야 그 위 모든 채점을 믿을 수 있다.

- 사람이 결과를 보고 `holdout.jsonl`을 확정(정답 채움 + `note`의 "추정" 제거). H12~14(거부)는 § 불필요.

**통과 기준**:

### [ ] 단계 H2: 검색 채점기 Recall·coverage·MRR [🔴수정4]

> 관심사: *"검색이 정답 §를 top-K로 물어왔나를 코드로 채점."*

#### 🟢 쉬운 설명 — 검색 채점기 [📄 깃헙 페이지스 참고]

**먼저 어려운 말:**
- **채점기(scorer)**: 검색이 정답을 잘 가져왔는지 *점수를 매기는 코드*. Claude 없이 공짜.
- **Recall@K(회수율)**: 정답 조항이 검색 상위 K개 안에 들어왔나.
- **coverage(커버리지)**: 정답이 여러 개일 때 *몇 개나* 가져왔나(비율).
- **MRR**: 정답이 *몇 등*으로 나왔나(1등일수록 높은 점수).

**비유: 장보기 심부름 채점 🛒** — 정답지에 **살 것 목록**(정답 §들)이 있고, 로봇이 마트에서 **K개를 담아온다**(검색 top-K). 세 가지로 채점한다.

**예시 — H10** (정답 § **2개**: §61.57 경험 + §61.23 의료), K=5로 담아온 게 `[§61.57, §91.103, §61.23, §71.1, §67.1]` 라면:

| 점수 | 질문 | H10 결과 | 뜻 |
|---|---|---|---|
| **Recall** | 목록에서 *하나라도* 담았나? | 1 (§61.57 있음) | 거칠다 — 절반만 맞아도 만점 |
| **coverage** | 목록 중 *몇 개* 담았나? | 2/2 = **1.0** | 정직 — 교차질문의 진짜 실력 |
| **MRR** | *첫 정답*이 몇 등? | §61.57이 1등 → **1.0** | 높을수록 앞에 = 토큰 절약·품질↑ |

> **왜 셋 다?** Recall만 보면 H10에서 §61.57 하나만 담아도 "만점"이라 거짓 신호([🔴수정4]). **coverage**가 "2개 중 몇 개?"를 정직하게 보고, **MRR**이 "정답이 1등이냐 8등이냐"를 봐 토큰·품질까지 챙긴다. → MRR이 높으면 K를 줄여도 돼 입력 토큰↓([실험4 최소-K]와 연결).

> **[R1] 채점은 청킹 축과 독립적으로.** 각 청크의 `section` 메타에만 의존하면 char-청킹(§태그 없음, `section=None`)이 조용히 recall=0으로 찍혀 *거짓 결론*을 낸다. 통일 규칙: **retrieved 청크가 정답 §를 "맞혔다"의 정의 = ①청크의 `section` 메타가 정답 §이거나, ②(메타 없으면) 청크 텍스트에 정답 § 번호 문자열이 있으면 hit.** 모든 청킹 방식에 같은 자를 적용.

**변경 내용**:
- `score.py`: `score_retrieval(hits: list[dict], expected: list[str], k) -> dict` (청크 dict를 받아 위 [R1] 규칙으로 §-hit 판정)
  - `recall` = 정답 § 중 하나라도 top-K에 있으면 1 else 0 (이진, 참고용)
  - `coverage` = |top-K가 맞힌 정답 §| ÷ |expected| (집합 커버리지 — 교차질문 정직)
  - `mrr` = 첫 정답 §의 순위 역수(1위=1.0, 5위=0.2, 없으면 0)
  - **[R5] 입력 정제**: retrieved §는 **dedupe + `None` 제거 + § 정규화**(공백/기호 통일) 후 판정. `expected`가 비면(거부문항) 채점 스킵(**0 나눗셈 가드**).
- `test_score.py`: ①단일정답 완전일치, ②교차질문 절반커버(coverage=0.5), ③정답 없음(0), ④MRR 순위 계산, **⑤[R1] char-청킹(section=None)인데 텍스트에 §번호 있으면 hit**, **⑥[R5] 중복 §·expected 빈 집합 가드**.

**통과 기준**:
- `pytest harness/tests/test_score.py -q` 통과(위 6케이스).
- H03 실측: §91.151 인덱스로 `search` → 채점기가 recall=1·coverage=1·mrr=1.0 산출.

**검증 통과 시 커밋**: `feat(harness): retrieval scorer — recall + set coverage + MRR with tests`

> 관심사: *"오케스트레이터가 축을 돌리려면, 인덱서·검색이 설정을 받아야 한다."* 지금은 둘 다 단일 목적이라 그리드를 못 돈다.

**변경 파일**: `rag-starter/indexer.py`(수정), `harness/retrieval.py`(신규)

**변경 내용**:
- **[R2]** `build_index(chunker, embed_model)` 매개변수화 — 지금 `MODEL_NAME`·청커 하드코딩을 인자로. (청킹 축: `1000자`/`§경계`/`라우팅`, 임베딩 축: MiniLM/bge/e5/gte)
  - **bge·e5는 쿼리 프리픽스 필수**(EXPERIMENTS §2), gte는 불필요 → 모델별 프리픽스 테이블.
- **[R3]** `retrieval.py`: 벡터전용 `search()` 위에 **BM25**·**하이브리드(α 가중 병합)** 추가. 인터페이스 통일: `retrieve(query, index, method, k, alpha) -> hits`.

**통과 기준**:
- `build_index("section","MiniLM")` 와 `build_index("char","MiniLM")` 가 각각 다른 청크 수로 빌드됨.
- `retrieve(q, idx, "bm25", 5)` / `"hybrid"` / `"vector"` 셋 다 hits 반환, 형식 동일.
- bge로 빌드 시 쿼리에 프리픽스가 실제로 붙는지 1케이스 확인.

**검증 통과 시 커밋**: `feat(harness): parameterize indexer (chunker/embed) + BM25/hybrid retrieval`

### [ ] 단계 H3b: 오케스트레이터 루프 [🔴수정2·R4·R8] — *엔진*

> 관심사: *"H3a 도구 위에서 설정을 순서대로 돌려 runs.jsonl을 쌓는 for문."* — 무료 검색 축만(0~4).

**변경 파일**: `harness/orchestrate.py`(신규), `harness/tests/test_orchestrate.py`(신규)

**변경 내용**:
- `orchestrate.py`: 중첩 루프 — **바깥**: (청킹 × 임베딩) → `build_index`(무거움, 12개), **안쪽**: (검색 × K) → 같은 인덱스 재사용(즉시).
  - 각 (설정 × holdout 질문) 실행마다 `score_retrieval` → `log_run(...)` 한 줄.
  - **[R4] 이어하기 키 = 설정 내용**(청킹+임베딩+검색+K+question_id), **`build_id` 아님**(매 실행 새로 생성돼 skip 무력화됨).
  - **비용가드**: 검색 축은 $0지만, 인덱스 빌드 횟수/시간 상한을 로그로. (유료 축은 범위 밖.)
  - **[실험4 = top-K 적정화]** K 스윕(3/5/8)에서 **coverage가 거의 안 떨어지는 최소 K**를 고른다([수정4]③, 교차질문 coverage 사수). *측정은 여기(무료), 토큰 절약 이득은 생성 단계에서 회수.* 항공법은 조항 1개로 답되는 게 많아 작은 K로도 충분한 경우가 많음.
- 이 단계는 STRATEGY 실험 0~4에 해당. finalist 2개는 2막/생성 단계로 넘김([수정3]).

**통과 기준**:
- 작은 그리드(예: 청킹1 × 임베딩1 × 검색2 × K2 = 4설정) × holdout 11문항 → `runs.jsonl`에 44줄 적재, 모두 유효 JSONL.
- 재실행 시 **[R4]** 설정키 이어하기로 **중복 줄 0**.
- **[R8]** `test_orchestrate.py`: 가짜 미니코퍼스(§ 2~3개)로 end-to-end 1회 — 빌드→검색→채점→runs.jsonl 적재 회귀 방지.
- 리더보드 원천으로 쓸 수 있게 config가 평면 필드로 인라인.

**검증 통과 시 커밋**: `feat(harness): grid orchestrator (index-build outer, retrieval/K sweep inner) with content-key resume`
